# 全生命周期知识复用平台需求规格说明书

**版本**：V2.0  
**拟定日期**：2026-04-19  
**变更摘要**：调整架构，取消项目硬隔离，新增跨项目索引与复用追踪机制  
**文档状态**：待评审

---

## 1. 项目背景与目标

### 1.1 背景
项目涉及论证、设计、实施、试验四个阶段，产生数据类型复杂，包括：文档、代码、二进制制品、STEP 三维图纸、试验视频/图片、试验报告等。

### 1.2 核心痛点
- 各阶段产物分散存储，追溯前因后果依赖人工记忆。
- 项目间成果复用率低，重复设计与论证现象普遍。
- 缺乏统一的技术状态基线，交付审计困难。
- 现有 Jira/禅道 无法管理非代码类大文件。

### 1.3 建设目标
构建一个**统一知识池**，实现：
1. **全生命周期追溯**：论证 → 设计 → 实施 → 试验，任意节点可双向追溯。
2. **跨项目复用**：打破项目边界，A 项目成果可直接被 B 项目引用。
3. **受控技术状态**：关键节点形成不可篡改的基线快照。
4. **全开源私有化**：所有组件基于开源协议，部署于企业内网。

## 2. 数据资产定义

| 数据类型 | 具体内容 | 单文件大小 | 格式 | 复用/检索要求 |
| :--- | :--- | :--- | :--- | :--- |
| **论证文档** | 需求条目、可行性分析、技术指标 | < 100 MB | Markdown/Word/PDF | 可被其他项目引用作为依据 |
| **设计图纸** | 三维模型 | < 4 GB | **STEP** | 可被其他项目复用，需追踪复用情况 |
| **制品** | 数据结构定义、编译二进制库、依赖包、配置文件 | < 4 GB | 混合 | 通用依赖库可跨项目共享 |
| **试验视频** | 试验过程录像 | < 4 GB | MP4/MOV | 可关联至对应论证条目 |
| **试验图片** | 试验现象截图 | < 50 MB | PNG/JPG | 可关联至对应论证条目 |
| **试验报告** | 数据分析、结论 | < 500 MB | Markdown/Word/PDF | 验证论证结论的关键证据 |
| **源代码** | 项目源码、脚本 | - | Git 托管 | 标准 Git 管理 |

## 3. 功能性需求

### 3.1 全局知识池架构

**FR-001 无项目硬隔离**
- 所有论证条目、设计图纸、试验数据均存入**统一知识池**。
- 系统**不设置项目间的访问边界**，任何已登录用户均可浏览、搜索所有项目资产。

**FR-002 项目归属标识**
- 每条数据需明确标注其**源项目名称**作为属性之一。
- 数据可按项目维度**过滤浏览**，但过滤不等于隔离。

**FR-003 全局唯一标识**
- 每一条论证条目、每一份图纸、每一份报告均拥有**系统内唯一 ID** 或 **永久 URL**，供跨项目引用使用。

---

### 3.2 关系型追溯与复用

**FR-004 双向链接机制**
- 支持在任意页面（论证条目、图纸说明、试验报告）中通过 **Wiki 语法** 或 **@ 提及** 引用其他资产。
- 引用关系需**双向可见**：
    - 正向：当前页面引用了哪些其他页面？
    - 反向：哪些页面引用了当前页面？

**FR-005 复用追踪**
- 当查看某份设计图纸时，系统需展示：
    - 该图纸当前被哪些项目引用。
    - 引用该图纸的项目列表及引用版本。

**FR-006 相似推荐**
- 当用户创建新论证条目或上传新图纸时，系统应能根据**标签**或**标题关键词**推荐知识池中已有的相似资产，提示复用。

**FR-007 引用版本锁定**
- 跨项目引用时，引用链接需**锁定引用时刻的版本**。
- 被引用方后续更新不自动同步至引用方，需引用方**手动确认更新**，以保证各项目技术状态独立可控。

---

### 3.3 技术状态与受控审批

**FR-008 工作状态可见性**
- 资产（文档/图纸/报告）上传后默认处于**工作状态（草稿）**。
- 工作状态的内容对全平台用户**可见**，并伴有明确标识（如 `[草稿]` 标签、灰色水印）。

**FR-009 审批受控流程**
- 支持将工作状态的资产提交**受控审批**。
- 审批流程：**提交审批 → 指定审批人确认 → 状态变更为“受控”**。
- 审批通过后，资产标记为 `[受控]`，内容进入**只读模式**。

**FR-010 不可更改的基线**
- 支持对项目在特定时间点的**全部受控资产集合**打基线（Snapshot）。
- 基线一旦生成，其包含的所有资产版本**永久不可篡改**，用于交付验收与审计。

**FR-011 版本历史**
- 所有资产的每次修改（含草稿阶段编辑）均需保留完整历史版本。
- 支持回滚至任意历史版本。

---

### 3.4 大文件存储管理

**FR-012 Git 代码托管**
- 源代码及配置文件使用 **Git** 管理，保留完整提交历史。

**FR-013 大文件专用存储**
- 二进制制品（库文件、依赖包）、STEP 图纸、试验视频/图片**不直接存入 Git 仓库**。
- 上述大文件存入**专用对象存储**，知识库页面中仅存储**引用指针**。
- 单文件上传上限不低于 **4GB**。
- 上传需支持**断点续传**。

**FR-014 文件下载**
- 大文件支持通过 Web 界面直接下载。
- 可选：支持生成临时分享链接，供外部合作方限时下载。

---

### 3.5 搜索与检索

**FR-015 全局全文搜索**
- 搜索范围默认覆盖**全平台所有资产**。
- 支持按以下维度过滤：
    - 源项目
    - 资产类型（论证/图纸/制品/试验视频/试验报告）
    - 状态（草稿/受控）
    - 标签

**FR-016 搜索结果展示**
- 每条搜索结果需清晰展示：
    - 标题
    - 源项目名称
    - 当前状态（草稿/受控）
    - 被引用次数（体现复用热度）

---

### 3.6 任务跟踪（Jira/禅道替代）

**FR-017 项目任务管理**
- 提供看板视图，支持创建、分配、追踪任务。
- 任务卡片支持关联知识池中的任意资产（通过 ID 或 URL 链接）。

**FR-018 任务与代码关联**
- 任务需支持关联 Git Commit 记录。
- 在任务详情页可查看关联的代码变更。

**FR-019 任务看板按项目分组**
- 任务看板按项目分组展示，保持操作聚焦。
- 但任务详情页中的**关联链接可跨项目指向**。

---

### 3.7 图纸特殊要求

**FR-020 STEP 格式存储**
- 图纸以 **STEP（.stp / .step）** 格式存储。
- **不要求在线预览**功能，仅需支持下载后本地查看。

**FR-021 图纸元数据**
- 上传图纸时需支持填写/编辑元数据：
    - 图纸名称
    - 版本号
    - 所属项目
    - 设计者
    - 关联论证条目（可选）
    - 标签（用于跨项目分类检索）

---

### 3.8 试验数据管理

**FR-022 试验视频与图片上传**
- 支持批量上传试验视频和图片。
- 支持通过 Web 界面或 WebDAV 协议拖拽上传。

**FR-023 试验报告编写**
- 支持在线编写试验报告（Markdown 或富文本）。
- 报告中可直接引用已上传的试验视频/图片。

**FR-024 试验与论证关联**
- 每份试验报告需明确关联其**验证的论证条目**。
- 论证条目页面需展示所有关联的试验报告及结论摘要。

## 4. 非功能性需求

### 4.1 部署与安全

**NFR-001 私有化部署**
- 所有服务端组件仅运行于企业内部服务器或私有云。
- **禁止**依赖任何公有云 SaaS 服务。

**NFR-002 全开源组件**
- 所有底层软件必须基于 **OSI 认证开源协议**（MIT、GPL、Apache 2.0 等）。

**NFR-003 容器化部署**
- 优先提供 Docker Compose 或 Kubernetes 部署方案，便于安装、升级、迁移。

**NFR-004 数据备份**
- 支持对知识库内容、Git 仓库、大文件存储进行**定期自动备份**。
- 备份数据需可独立恢复。

---

### 4.2 性能指标

**NFR-005 并发用户**
- 支持 **50** 并发用户同时访问，响应时间 < 2 秒。

**NFR-006 大文件传输**
- 4GB 单文件上传支持**断点续传**。
- 下载速度受限于内网带宽，不做额外限速。

**NFR-007 存储扩展**
- 大文件存储节点支持横向扩展，满足视频数据长期增长需求。

---

### 4.3 易用性

**NFR-008 低学习成本**
- 界面操作逻辑对标主流 Wiki（如 Confluence）或网盘（如 Nextcloud）。
- 提供 WebDAV 支持，用户可将知识库映射为本地网络驱动器。

**NFR-009 链接即引用**
- 跨页面引用操作应**无需跳出当前编辑界面**，通过 `[[` 或 `@` 快捷键触发搜索并插入链接。

## 5. 推荐技术架构方案

### 5.1 组件选型矩阵

| 功能模块 | 推荐组件 | 选型理由 |
| :--- | :--- | :--- |
| **知识库核心** | **DokuWiki** | 平文件存储，无需数据库；原生支持命名空间（项目）与标签；支持双向链接插件 |
| **大文件存储** | **MinIO** | 开源 S3 兼容对象存储，单文件 4GB+ 无压力，支持断点续传 |
| **代码托管** | **Gitea** | 轻量 Git 服务，资源占用低，内置 Issue 跟踪与 Wiki |
| **任务看板** | **Plane** | 现代化项目管理工具，UI 接近 Linear，支持与 GitHub/Gitea 集成 |
| **试验数据网盘** | **Nextcloud** | 原生支持 WebDAV，视频/图片上传体验好，可与 OnlyOffice 集成编辑报告 |
| **反向链接/知识图谱** | DokuWiki 插件：`tag` + `backlinks` | 实现 FR-004 / FR-005 要求的复用追踪 |
| **审批工作流** | DokuWiki 插件：`publish` 或定制脚本 | 实现 FR-009 要求的受控审批状态变更 |
| **全局搜索** | DokuWiki 内置搜索引擎 | 全文索引速度快，支持页面内容与元数据检索 |

### 5.3 数据流向说明

1. **论证与设计文档** → 写入 **DokuWiki**，通过 `[[项目:页面名]]` 语法建立双向链接。
2. **STEP 图纸上传** → 前端上传至 **MinIO**，Wiki 页面中插入 MinIO 文件链接，并标注元数据。
3. **试验视频/图片** → 可通过 **Nextcloud WebDAV** 批量拖拽上传，实际文件落盘于 **MinIO**。
4. **代码与制品** → 代码推送到 **Gitea**；大二进制制品通过 Gitea LFS 推送至 **MinIO**。
5. **任务管理** → **Plane** 中创建任务，关联 Gitea Issue/PR 及 DokuWiki 页面 URL。

## 6. 术语表

| 术语 | 定义 |
| :--- | :--- |
| **知识池** | 不设项目边界的统一数据存储空间，所有项目资产共享于此 |
| **双向链接** | 指 A 引用 B 的同时，B 页面自动显示“被 A 引用”的链接列表 |
| **受控状态** | 内容经审批确认，不可随意修改的状态，需走变更流程方可更新 |
| **基线** | 特定时间点所有受控资产版本的不可变快照 |
| **制品** | 编译后的二进制库、可执行文件及依赖包，区别于源代码 |
| **STEP** | ISO 10303 标准三维模型交换格式 |
| **WebDAV** | 允许将远程服务器映射为本地磁盘的协议 |
